# Мой Сайт — Внешняя память проекта

## Технологический стек
- **Backend**: Python (Flask)
- **Frontend**: HTML5, Vanilla CSS, Vanilla JavaScript
- **Стилизация**: Чистый CSS без фреймворков

## Структура проекта
- `/templates/` — HTML-шаблоны
- `/static/`
  - `/css/` — файлы стилей (например, `style.css`)
  - `/js/` — скрипты (например, `main.js`)
- `/References/` — макеты и референсы (кнопки, шапка, контент)
- `app.py` — главный файл Flask-приложения
- `requirements.txt` — зависимости проекта
- `AGENTS.md` — этот файл (внешняя память)

## Команды запуска и разработки
- Виртуальное окружение: `.venv`
- Запуск приложения: `.venv/bin/python app.py` (запущено на порту 5000 в режиме отладки `debug=True`)

## Принятые технические решения
1. **Анализ референсов кнопок (Normal)**:
   - Исходный цвет кнопки: `#ebebeb` (серый).
   - Цвет при наведении (Hover) и выбранном состоянии (Selected): `#b3f7ff` (циановый).
   - Цвет текста и линий: `#000000` (черный).
   - Кнопка имеет размеры примерно 300x80px (в исходном варианте, но в шапке кнопки пропорционально уменьшены для аккуратного внешнего вида).
2. **Анимация кнопки**:
   - Переход рамки из уголков и плюсика в сплошную линию реализуется за счет расширения псевдоэлементов/абсолютных блоков `.corner-tl`, `.corner-br` (до `100%` ширины и высоты) и втягивания/роста линий плюсика `.plus-line-v` и `.plus-line-h` при ховере или в классе `.selected`.
   - Длина усиков плюсика и уголков в исходном состоянии установлена в `20px` со всех сторон (для обеспечения визуальной симметрии).
   - Чтобы передняя серая панель кнопки `.btn-body` не перекрывала внутренние усики плюсика и уголков в исходном и ховер состояниях, фон переднего слоя в этих состояниях сделан прозрачным (`background-color: transparent`). Серый или циановый фон в этих состояниях рендерится исключительно на заднем слое `.btn-shadow`.
   - При клике передний слой кнопки `.btn-body` становится непрозрачным (циановым) и сдвигается по `transform: translate(12px, -12px)` (или меньшему значению в зависимости от размера кнопки), в то время как задний слой `.btn-shadow` с рамкой остается неподвижным и возвращает серый цвет `#ebebeb`.
3. **Анимация и сжатие шапки**:
   - Шапка (`.site-header`) имеет два состояния по высоте: сжатое (`60px` по умолчанию) и раскрытое (`100px` при `:hover`). Это экономит экранное пространство и предотвращает перекрытие основного контента, когда курсор находится вне верхней зоны.
   - Круг логотипа (`.logo-circle`) и текст («Лого») плавно масштабируются синхронно с высотой шапки за счет CSS-переходов.
4. **Адаптивность элементов шапки**:
   - Вместо скрытия элементов в бургер-меню (которое отложено), кнопки остаются в горизонтальном флекс-контейнере и пропорционально уменьшаются (до `100px` на ноутбуках и до `80px` на мобильных экранах в покое) при переходе на меньшие разрешения (медиа-запросы `< 1440px` и `< 768px`). Декоративные элементы кнопок (вылеты плюсиков, длина усиков уголков) переопределяются в CSS для сохранения точных визуальных пропорций.
5. **Многостраничность и навигация**:
   - Маршрутизация на бэкенде: Flask обрабатывает роуты `/` (Главная страница) и `/resume` (Резюме).
   - Навигационные элементы в шапке переопределены как теги `<a>` с прямыми ссылками на соответствующие разделы.
   - Разделение функционала: боковое меню «Содержание» отображается только на странице Резюме.
   - Плавная прокрутка: на странице Резюме клики на пункты бокового меню плавно прокручивают страницу к секциям резюме с помощью связки `data-target` атрибутов и JS-метода `scrollIntoView()`.

- [x] Шаг 15: Текст наезжает на bevel при скролле дерева — перенос padding с скролл-контейнера на рамку + z-index изоляция.
  - **Причина (Шаг 14 был недостаточен)**: Шаг 14 разделил frame/scroll, но padding остался на скролл-контейнере (`.explorer-tree { padding:15px 10px }`, `.explorer-scroll { padding:30px }`). Скролл-контейнер клиппит по padding-box (= border edge), поэтому текст при скролле проходит через padding-область (0..P px) и виден в зоне 3..P px рядом с bevel — воспринимается как «наезд на границу-тень». Bevel `::after` z-index:2 действительно поверх контента (доказано hit-test перехватом при `pointer-events:auto`), но текст в padding-области ВИДЕН ниже bevel-полосы (3..P px) — это и есть «наезд».
  - **Решение (чистое разделение по просьбе пользователя)**: padding перенесён с скролл-контейнера на **рамку**. Между bevel и текстом — чистый белый отступ рамки (padding-area, показывает bg рамки).
  - **CSS (base)**: `.explorer-sidebar { padding: 15px 10px }` (было 0), `.explorer-content { padding: 30px }` (было 0) — padding на рамках. `.explorer-tree { padding: 0; position: relative; z-index: 1 }` (было padding 15px 10px), `.explorer-scroll { padding: 0; position: relative; z-index: 1 }` (было padding 30px) — скролл-контейнеры без padding, изолированы в собственный stacking context НИЖЕ bevel (`::after` z-index:2). Двойная защита: текст клиппит до bevel + bevel поверх SC контента.
  - **CSS (responsive)**: 1440 — `.explorer-content { padding: 20px }` + `.explorer-scroll { padding: 0 }` (было наоборот). 768 — `.explorer-sidebar { padding: 10px 5px }` + `.explorer-tree { padding: 0 }`, `.explorer-content { padding: 15px }` + `.explorer-scroll { padding: 0 }` (было наоборот).
